Basic Concepts

Q Light Controller (QLC for short) is meant to control lighting equipment used in various performances, like live concerts and theatres etc. The main purpose is that QLC could outperform commercial lighting desks with an intuitive and flexible user interface, without the need for a 500+ page manual.

Modes

Q Light Controller is based on the common concept of having two distinct operational modes to prevent accidental or otherwise harmful changes during operation. Design mode is meant to edit the behaviour of the program; create and edit functions and fixtures and how they work. Operate mode is meant to execute the created functions that eventually control your lighting fixtures.

Output

DMX output is achieved thru output plugins that can support one or more physical DMX universes. It is also possible to run QLC without actual DMX hardware.

Input

Primary input methods for QLC are naturally the keyboard and mouse. Users can attach keyboard keys to virtual console buttons and drag sliders and do pretty much everything with a mouse.

Special input plugins provide capabilities for getting external devices (mainly slider boards) to provide input data to QLC elements. These input devices can be attached to virtual console widgets to eliminate the need for controlling a lighting show with a clumsy mouse or keyboard. External input can be used also to edit some function contents.

Fixtures

The software is based on the DMX512 standard, which defines the number of channels and their value range, as well as other protocol-related information. QLC supports 4 individual DMX universes, each consisting of 512 channels, each channel having an integer value between 0 and 255.

Users can store shared fixture information in a fixture library that contains the following properties for each fixture:

  • Manufacturer (e.g. Martin)
  • Model (e.g. MAC250)
  • Type (Color Changer, Scanner, Moving Head, Smoke, Haze, Fan...)
  • Channels:
    • Channel group (Pan, Tilt, Gobo, Color, Speed etc.)
    • 8bit and 16bit channel bindings for pan & tilt groups
    • Value ranges for channel features (e.g. 0-5:Lamp on, 6-15:Strobe etc..)

    These fixture entries can then be used to create fixture instances (256 at most at a time), that will, after instantiation, have additional properties defined by users:

  • DMX Universe
  • DMX Address
  • Name

    • Several instances of a fixture can be created (i.e. users must be able to have several instances of a MAC250 in use). Each fixture can be named, but the name is not used internally to identify individual fixture instances. The same goes for the DMX address. Nevertheless users are encouraged to name their fixtures in some systematic way to help identify each of them -- if necessary.

    Generic dimmer devices don't need their own fixture definitions, because usually multiple dimmers are patched into a common address space, employing one or more dimmers. Users can create instances of these generic dimmer entities just by defining the number of channels each one of them should use.

    Functions

    Users can create 4096 functions, that are used to automate the setting of values to channels. The different function types are:

  • Scene
  • Chaser
  • EFX
  • Collection

    • Each function can be named, but the name is not used internally to identify individual functions. Nevertheless users are encouraged to name their functions in some systematic way to help identify each of them -- if necessary.

    Speed Setting (buses)

    Functions have a speed setting that is set by users. A maximum of 32 different speed settings called buses, are possible. One function can have exactly one bus and many functions can share one bus. The speed settings can be changed at any time with a resolution of 1/64 seconds. Any of the buses can be assigned in design mode to any function (except collection-type functions, which have no meaning for a speed). A name can be assigned to each of the buses, for convenience and to help identification.

    The names of the first two buses are, by default, "Fade" and "Hold". Functions' completion times and their run-time accuracy in value setting can be late or early 10 milliseconds on average. The user is able to quickly stop all running functions (pressing a panic button) in case something disastrous happens.

    Scene

    A scene function encapsulates the values of selected channels owned by one or more fixture instances. When a scene is started, the duration it takes for the values to reach their target depends on the function's speed setting (see the section on Speed Setting above). Bus number 1 (Default Fade) is assigned to all newly created scenes by default. If the bus' value is 0 seconds, the scene values are set immediately and no fading occurs. Otherwise the values are always faded from what they currently are, to the target values defined in the scene. Channels that are not enabled in the scene will not be touched.

    Copies of scene functions can be created within the 4096 total function limit. All of the function's contents are copied to the duplicate. After copying, the two functions are completely independent entities that behave exactly the same.

    EFX

    An EFX function is mainly used to automate moving lights (e.g. scanners & moving heads). The EFX can create complex mathematical paths on an X-Y plane that are converted to DMX values for the fixture's pan and tilt channels. Only fixtures that contain valid pan & tilt channels can take part in an EFX function.

    Bus number 1 (Default Fade) is assigned to all newly created EFX's by default. The bus value is used as the overall time for the EFX algorithm to complete one cycle. In other words, with a circle algorithm, the bus value is understood as the time available to complete a full 360 degree circle.

    EFX functions can have special start and stop scenes, that are run just before the EFX is started and immediately after the EFX stops. These scenes are not, however, faded as would happen with a normal scene, but instead, they are set immediately, as if the speed set thru their bus was set to zero.

    Fixtures taking part in an EFX function can be set to follow the algorithm either in parallel (all fixtures following the same pattern synchronously) or in serial (fixtures start following the pattern one after another, with a little delay in-between, creating a more lively, wave-like effect).

    EFX functions' direction can be reversed for all fixtures at once or on a per-fixture basis. The function can also be set to do an infinte loop, an infinite ping-pong-loop (direction is reversed after each pass) or it can run thru just once, in a single-shot mode, after which it terminates by itself. If the function is set to do an infinite loop, it must be stopped manually.

    Copies of EFX functions can be created within the 4096 total function limit. All of the function's contents are copied to the duplicate. After copying, the two functions are completely independent entities that behave exactly the same.

    Chaser

    A chaser function encapsulates multiple scenes that are executed in serial one after the other when the chaser function is run. The next function is run after the previous has finished. Any number of scenes can be inserted to a chaser.

    Chaser functions' direction can be reversed. The function can also be set to do an infinite loop, an infinite ping-pong-loop (direction is reversed after each pass) or it can run thru just once, in a single-shot mode, after which it terminates by itself. If the function is set to do an infinite loop, it must be stopped manually.

    Bus number 2 (Default Hold) is assigned to all newly created chasers by default. The bus value is used as hold time between steps. Fade time for the chaser's member scene functions is set thru the scene function's own speed buses.

    Copies of chaser functions can be created within the 4096 total function limit. All of the function's contents are copied to the duplicate. After copying, the two functions are completely independent entities that behave exactly the same.

    Collection

    A collection function encapsulates multiple functions that are executed simultaneously when the collection function is executed. Any number of functions can be inserted to a collection. One function can be inserted only once and a collection cannot be a member of itself.

    Speed setting has no effect on a collection. The collection's functions' speeds are set individually by its member functions' buses.

    Copies of collection functions can be created within the 4096 total function limit. All of the function's contents are copied to the duplicate. After copying, the two functions are completely independent entities that behave exactly the same.

    Virtual Console

    The purpose of the virtual console is to act as an open sandbox where the user can create a lighting desk layout of his choice. Users can create various GUI (Graphical User Interface) elements, called widgets, onto the console surface:

  • Buttons for starting and stopping functions
  • Sliders for adjusting selected bus or channel values
  • XY Pads for manually moving intelligent light beams
  • Cue lists for theatrical performance purposes
  • Frames for grouping various widgets together
  • Labels to act as static information banners for other widgets
