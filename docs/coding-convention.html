Q Light Controller 2 Coding Convention

This file describes the coding convention used in qlc source code.
The purpose of this document is to instruct developers to write somewhat
uniform code. The following rules are more or less similar to Qt's coding
convention.

2. Each member variable has a prefix "m_", for example int m_fooMember;
Each global variable or function begins with "_", for example int _foo;

3. Each function begins with a small letter and in compound words, all the following words begin with a capital letter. No word separation with "_" is allowed. Examples: 

   int foo();
   int fooBar();
   int thisIsAMemberFunction();
   int this_is_not_allowed_because_it_is_ugly_and_differs_from_Qt_convention();

4. Every control structure (if, while, for...) will be written inside brackets { } even if they would contain only one line of code. Also, the brackets are written on a new line without anything else after the bracket. This goes for functions, too: 

   do
   {
	foo();
   }
   while (bar);

   if (foo == true)
   {
	foo();
   }
   else
   {
	bar();
   }

   for (int i = 0; i < n; i++)
   {
	foo(i);
   }

   void FooClass::fooBar()
   {
	...
   }

5. Each class name will be written each word beginning with a capital letter: 

   class FooBar;
   class Foo;

5.1. Respectively, each class should be in its own file and the file name must be
the class' name, in small letters. If it makes things significantly more logical,
classes can be defined inside other classes' definitions and files. File suffix for
implementation files is ".cpp" and for header files ".h". Do NOT use other styles.
In general:

   class FooBarClass definition is saved in foobarclass.h
     and its implementation is in foobarclass.cpp

6. All macro definitions are written completely in capital letters, words may be separated with "_". Example: 

      #define FOOBAR 5
      #define FOO_BAR(x) foobar(x);

7. Each operator will be separated with spaces: 

   if (foo == bar)
   while ( foo != bar)
   int foo = bar;

8. Variable, function and class names should be descriptive. Use your common sense here.

9. Do not write code in one pile, use spaces and empty lines between logical sections.

10. Write nice and readable code.

11. WRITE COMMENTS. I am a BAD example on this...

12. Mark each code modification you make with a comment containing your sourceforge user name, date (format dd-mmm-yyyy) and a brief description. 

    // hjunnila 27-Jan-2002
    // foo should be bar, not foobar
    foo = bar;
